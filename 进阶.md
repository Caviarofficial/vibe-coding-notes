# 常见的工作流程
1. 通用流程，这种多功能工作流程适合许多问题（**探索、规划、编码、提交**）：​
   1. 要求Claude**阅读相关文件、图像或URL**，提供一般指引（"阅读处理日志记录的文件"）或具体文件名（"阅读logging.py"），但明确告诉它**暂时不要编写任何代码**。​
   2. 要求Claude**制定**解决特定问题的**计划**。我们建议使用"think"一词来触发扩展思维模式，这为Claude提供额外的计算时间来更彻底地评估替代方案。这些特定短语直接映射到系统中递增的思维预算级别："think" < "think hard" < "think harder" < "ultrathink"。每个级别为Claude分配越来越多的思维预算。​
   3. 要求Claude在代码中**实施其解决方案**。这也是一个可以要求它在实施解决方案的各个部分时明确验证其解决方案的合理性的好地方。​
   4. 要求Claude**提交结果**并创建拉取请求。如果有需要，这也是让Claude更新任何README或变更日志以说明它刚才所做工作的好时机
> 第1-2步至关重要 —— 没有它们，Claude倾向于直接跳到编码解决方案。虽然有时这正是您想要的，但要求Claude首先进行研究和规划显著改善了需要深度前期思考的问题的性能
2. 基于预期输入输出的工作流程（**编写测试，提交；编码，迭代，提交**）
   >这是Anthropic最喜欢的工作流程，用于可以通过单元测试、集成测试或端到端测试轻松验证的更改
   1. 要求Claude基于预期的输入/输出对编写测试。明确说明您正在进行测试驱动开发，这样它就会避免创建mock实现，即使对于代码库中尚不存在的功能也是如此。​
   2. 告诉Claude运行测试并确认它们失败。明确告诉它在这个阶段不要编写任何实现代码通常是有帮助的。​
   3. 要求Claude在您满意时把测试提交到git。​
   4. 要求Claude编写通过测试的代码，指示它不要修改测试。告诉Claude继续直到所有测试通过。Claude通常需要几次迭代来编写代码、运行测试、调整代码并再次运行测试。​
   5. 在您满意更改后要求Claude提交代码到git
3. 有视觉参与的工作流程（**编写代码，截图结果，迭代**）
   1. 为Claude提供**拍摄浏览器截图的方法**（例如，使用Puppeteer MCP服务器、iOS模拟器MCP服务器，或手动==复制/粘贴截图到Claude中==）。​
   > 比如在网上看到了特别欣赏的网页、PPT页面、设计稿，可以截图丢进去，用MCP实现
   2. 通过复制/粘贴或拖放图像为Claude提供**视觉输入**，或给Claude图像文件路径。​
   3. 要求Claude在代码中**实现设计**，拍摄结果截图，并**迭代直到其结果与模拟匹配**。​
   4. 在您满意时要求Claude提交。

# 对现有代码进行问答
在接触新代码库时，使用Claude Code进行学习和探索。您可以向Claude提出与其他项目工程师进行结对编程时相同类型的问题。Claude可以智能地搜索代码库来回答一般性问题，例如：
1. 日志记录是如何工作的？​
2. 我如何创建一个新的API端点？​
3. foo.rs第134行的async move { ... }做什么？​
4. CustomerOnboardingFlowImpl处理哪些边缘情况？​
5. 为什么我们在第333行调用foo()而不是bar()？​
6. baz.py第334行在Java中的等价物是什么？

# 与github交互
1. 连接：
   1. 用命令winget install --id GitHub.cli下好CLI
   2. 查看版本看看是否安装成功：gh --version
   3. 用gh auth login登录
   >登录方式：GitHub.com,协议：HTTPS,认证方式：建议选 Login with a web browser（最方便）,是否使用 SSH：通常选 No（除非你配置过 SSH key）
2. 交互例子：
   1. 创建拉取请求：Claude理解"pr"简写，并将根据差异和周围上下文生成适当的提交消息。​
   2. 实施简单代码审查评论的一次性解决方案：只需告诉它修复PR上的评论（或者也可以给它更具体的指令），完成后推送回PR分支。​
   3. 修复失败的构建或代码检查警告​。
4. 分类和整理Open的Issues，要求Claude循环遍历开放的GitHub Issues。
> 这让开发者无需记忆如何使用gh命令行，也能自动化进行常规任务

# 创建agent skill
1. skill分为两种：
   1. 存放全局skill的文件夹:**~/.claude/skills/**
      1. 具体路径为：C:\Users\你的用户名\.claude，
      2. 特点：对**电脑上所有项目生效**
      3. 存放个人通用 Skills、全局设置
   2. 存放某个特定项目的文件夹：**./.claude/skills/**
      1. 如果目前在文件夹D:\projects\my-app\下，那么这个文件夹的路径就是D:\projects\my-app\.claude\skills\
      2. 特点：仅目前文件夹对应项目（这里为my-app）可用的 Skills
      3. 存放项目特定的 Skills、规则，可以通过 Git 和团队共享
2. md中包含模块：
   1. 名字与描述，名字必须与skill文件夹名一致，描述用中文即可,claude就是通过描述去调用的
   2. 具体要求部分，这一部分与上面一部分用分割线隔开
3. 创建法：
   1. 在当前项目文件夹中终端执行命令**mkdir -p ~/.claude/skills/skill文件夹名（这里为my-app代替）**或者**mkdir -p ./.claude/skills/skill文件夹名**
   2. 在my-app中先创建文件**skill.md**在其中按相似格式填入要求，见./skill示例.md
4. 使用法：
   1. 被动调用：  
      1. 输入/skills回车（别忘末尾-s），如果检测到之前写好的就会显示在列表中
      2. 在输入框中提出具体要求，要求中可以提到调用某个特定的skill也可以不提，回车之后如果它发现请求与某个skill相关，就会调弹出请求允许
   2. 主动调用：请求行写上** /skill文件夹名 请求具体内容 **，则必然调用这个特定的skill
   >如：  /daily-report 写一份每日总结

# sub agnet
1. 含义：从属于claude的，有自己的上下文与skill的智能体
2. 创建：
   1. 输入/agent
   2. 选择create new agent
   3. 选择Project（含义为为项目而非用户创建agent）
   4. 选择基于claude创建：Generate with Claude (recommended)
   5. 在描述中用中文描述这个agent的**功能**与**调用场景**，如：**这是一个用于代码审核的agent，在用户要求“代码审核”的时候调用它**
   6. 接下来会选择给它使用的工具，由于是代码审核agent，选择只读工具Read-only tools即可，模型使用默认的sonet
   7. 为agent挑选颜色
   8. 选择一个应用范围范围，视情况而定，最好是推荐的Project scope (.claude/agent-memory/) (Recommended)
   9. 展示出第一版的subanget的描述，按**E**进入检查编辑，它自己写的是英文而且不太可行，一般在这一步就替换成一些大佬验证的比较可信的agent描述
   10. 创建好了之后重启claude
3. 与skill的一些区别
   1. agent skill：共享主对话上下文，会调用上下文的所有数据，适合处理**与上下文关联比较大，对上下文编码内容影响不大的任务**，如根据今天开发过程做每日总结
   2. subagent:拥有完全独立的上下文，启动时会**开辟全新的对话窗口**，此对话与主对话互不影响，所有分析结果都不传入主对话，只有活全部结束才会拿最终结果向我汇报，适合处理**与上下文关联比较小，对上下文编码内容影响大的任务**，比如审核代码
